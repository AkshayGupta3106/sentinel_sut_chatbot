# MLOps and CI/CD for Machine Learning

MLOps extends traditional DevOps practices (continuous integration, continuous delivery, infrastructure as code) to the additional complexity ML systems introduce: not just code changes, but data changes and model changes, each of which can independently break a system in ways traditional software CI/CD was never designed to catch.

Traditional CI/CD tests code against a fixed set of unit and integration tests and, if they pass, deploys it. ML CI/CD needs to additionally validate data (schema checks, distribution checks against a reference dataset), validate the model itself (offline metrics against a held-out test set, comparison against the currently deployed model's metrics, fairness/bias checks on relevant slices), and only then proceed to a staged deployment (shadow, canary, A/B — see ab_testing_deployment.md) rather than a direct full rollout, since offline metrics passing doesn't guarantee production behavior is safe.

Model versioning needs to track more than code: the training data snapshot (or a pointer/hash identifying exactly which data version was used), the exact hyperparameters, the training code version, and the resulting model artifact, all linked together so that any production model's lineage can be fully reconstructed for debugging or compliance purposes. Tools like MLflow, Weights & Biases, and DVC (Data Version Control) exist specifically to solve this multi-artifact versioning problem that Git alone wasn't designed for.

Reproducibility is harder in ML than traditional software because of non-determinism sources traditional CI/CD doesn't have to think about: random weight initialization, data shuffling order, GPU floating-point non-determinism, and library version drift can all cause the "same" training run to produce a measurably different model. Pinning random seeds, logging exact library versions, and containerizing the training environment all help, but perfect bit-for-bit reproducibility is often not fully achievable with GPU training and is not always worth the engineering cost to chase completely.

Rollback strategy needs to be planned before an incident, not during one: because a bad model deployment often doesn't throw an exception (it just makes systematically worse predictions), automated rollback triggers typically watch business/quality metrics (conversion rate, error rate, a proxy quality score) rather than relying solely on infrastructure-level health checks, which would show a bad model as perfectly healthy since it's still returning valid HTTP 200 responses.

A common interview question: "How is testing a machine learning system fundamentally different from testing traditional software?" Answer: traditional software tests specify exact expected outputs for given inputs and are inherently deterministic; ML systems require statistical tests instead — is accuracy on a held-out set above a threshold, has the prediction distribution shifted beyond an acceptable bound, does a fairness metric on a specific subgroup stay within range — because there generally isn't one single "correct" output per input to assert against, and the system's correctness is a property of its aggregate statistical behavior rather than any individual prediction.
