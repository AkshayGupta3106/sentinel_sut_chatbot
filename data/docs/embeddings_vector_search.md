# Embeddings and Vector Search

An embedding is a dense, fixed-length vector representation of an input (text, image, audio) learned such that semantically similar inputs end up close together in the vector space, measured by a distance metric like cosine similarity or Euclidean distance. This is the foundational representation that makes similarity search possible at all — comparing raw text character-by-character can't capture that "car" and "automobile" mean nearly the same thing, but their embeddings will be close together.

Sentence/text embedding models (e.g., sentence-transformers, OpenAI's or Gemini's embedding APIs) are typically trained with contrastive learning objectives — pulling embeddings of semantically similar text pairs closer together while pushing dissimilar pairs apart — often using techniques like triplet loss or in-batch negative sampling, so that the geometry of the resulting vector space actually reflects meaning rather than superficial lexical overlap.

Exact nearest-neighbor search (comparing a query vector against every single stored vector) is accurate but scales linearly with dataset size, becoming impractically slow once a collection reaches millions of vectors. Approximate Nearest Neighbor (ANN) algorithms trade a small amount of accuracy for large speed gains. HNSW (Hierarchical Navigable Small World graphs) builds a multi-layer graph structure enabling fast traversal toward nearest neighbors and is the default index type in most modern vector databases (including ChromaDB) due to its strong recall/speed tradeoff. IVF (Inverted File Index) partitions the vector space into clusters and only searches the most promising clusters for a given query, trading some recall for large speed gains at very large scale.

Chunking strategy for retrieval-augmented generation is a frequently underrated design decision: chunks that are too large dilute relevance (a chunk about five different subtopics matches many queries poorly) and waste context window budget, while chunks that are too small lose necessary surrounding context (a sentence fragment without the paragraph that explains it). Common strategies include fixed-size chunking with overlap (simple, but can cut sentences awkwardly), sentence/paragraph-boundary-aware chunking (respects semantic boundaries), and semantic chunking (grouping content by topic shift, detected via embedding similarity between adjacent segments).

Hybrid search combines dense vector similarity with traditional sparse lexical search (like BM25/TF-IDF), since pure embedding search can miss exact keyword matches (e.g., a specific product SKU or an acronym) that a sparse method catches trivially, while pure lexical search misses semantic paraphrases that embeddings catch easily. Combining both, often via a weighted score fusion or a re-ranking step, tends to outperform either approach alone in production RAG systems.

A common interview question: "Your RAG system's retrieval quality dropped after you switched to a cheaper embedding model. How would you confirm this is actually the cause, and what would you check next?" Answer: run the exact same golden query set through both old and new embedding models and compare hit-rate/precision directly — if the cheaper model shows measurably lower recall on the same golden set, that's confirmation, not just correlation with the deployment timing; from there, check whether chunk size or hybrid search (adding sparse lexical matching back in) can recover most of the lost quality before deciding whether the cost savings are actually worth it.
