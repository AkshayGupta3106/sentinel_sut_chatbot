# Continual Learning

Continual learning (also called online learning or incremental learning in various contexts) is the practice of updating a deployed model with new data on an ongoing basis, rather than retraining from scratch on a fixed dataset each time. It exists to solve the same problem data drift creates: a static model trained once and never updated gets progressively worse as the world it operates in changes.

Catastrophic forgetting is the central technical challenge: when a neural network is updated on new data, gradient descent has no inherent mechanism to preserve what it learned from old data, and performance on old patterns can collapse even while performance on new patterns improves. This is fundamentally different from traditional software updates, where adding a new feature doesn't usually break old features.

Common mitigation strategies include rehearsal (mixing a sample of old training data back in with new data during each update, so the model keeps seeing examples of what it shouldn't forget), regularization-based approaches like Elastic Weight Consolidation (which penalizes changes to weights that were important for old tasks, identified via their estimated importance/Fisher information), and architectural approaches that allocate separate capacity (extra parameters or modules) for new tasks while freezing what was learned before.

There's an important distinction between model iteration (the traditional cycle: collect data, retrain from scratch periodically, evaluate offline, deploy a new full model version) and true continual/online learning (the model updates incrementally, sometimes on every new labeled example, without a full retrain). Most production ML systems today actually use frequent full retraining on a rolling data window rather than true online learning, because it's operationally simpler and far easier to debug and roll back than a model whose weights are continuously mutating in production.

The evaluation challenge is distinct too: a continually updating model needs online evaluation (comparing predictions to labels as they arrive, monitoring for degradation in near-real-time) since there's no fixed held-out test set that stays representative for long — the model and the data distribution are both moving targets simultaneously.

A common interview question: "Why not just retrain your model every single time new labeled data arrives, rather than on a daily/weekly batch schedule?" Answer: cost and stability. Continuous per-example updates risk both catastrophic forgetting and unstable, hard-to-debug production behavior, and most business use cases don't actually need minute-by-minute freshness — a well-chosen retraining cadence (matched to how fast the underlying data distribution actually drifts) captures most of the benefit at a fraction of the engineering and monitoring complexity of true online learning.
