# A/B Testing and Deployment Strategies for ML Models

Shipping a new model version safely is a distinct engineering problem from training a good model — a model that scores better offline can still regress key business metrics online, and deployment strategy is how teams catch that before it causes damage at full scale.

Shadow deployment (also called dark launch) runs the new model alongside the current production model on live traffic, but only the old model's predictions are actually served to users — the new model's predictions are logged for comparison but never acted on. This is the safest way to validate a new model's behavior on real traffic patterns with zero user-facing risk, at the cost of not being able to measure actual downstream impact (since the new model's predictions never influence real outcomes).

Canary deployment routes a small percentage of real traffic (e.g., 1-5%) to the new model and monitors its metrics closely before gradually increasing that percentage. Unlike shadow deployment, the canary's predictions are actually served and do affect real outcomes, so it can measure true business impact, but it also carries real (if limited) risk if the new model is genuinely worse.

A/B testing splits traffic between model versions (typically 50/50, though canary-style ramps are common too) and uses statistical hypothesis testing to determine whether the difference in a target metric (click-through rate, conversion, revenue per user) is statistically significant rather than random noise. Getting this right requires care around sample size (underpowered tests can't detect real effects), the multiple comparisons problem (testing many metrics simultaneously inflates false positive rate), and network effects (in social/marketplace products, treating one user can affect another user's outcomes, violating the independence assumption most A/B testing math relies on).

Feature flags and gradual rollout let a team decouple code deployment from actual feature activation: the new model's serving code can be deployed to production while still fully disabled, then turned on for an internal team, then a small user segment, then everyone — with the ability to instantly roll back by flipping the flag rather than needing an emergency redeploy.

A common interview question: "Your canary at 5% traffic shows a new model improving CTR by 2%, but it's not statistically significant yet. What do you do?" Answer: it depends on the cost of waiting versus the cost of a wrong decision — options include increasing the canary percentage to get a larger sample faster, running the test longer, or using a sequential testing method designed to reach valid conclusions with less fixed sample-size waiting, rather than just eyeballing an underpowered result and shipping anyway.
