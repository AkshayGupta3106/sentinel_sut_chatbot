# Data Engineering Fundamentals for ML Systems

Most of the engineering effort in a real ML system goes into data, not modeling — a commonly cited rule of thumb is that data collection, cleaning, and pipeline engineering consume the majority of an ML project's time, with model training itself being a comparatively small fraction.

Structured data (relational tables with a fixed schema), semi-structured data (JSON, XML, log lines with nested but flexible fields), and unstructured data (raw text, images, audio) each demand different storage and processing approaches. A production ML system for something like a RAG chatbot typically touches all three: structured metadata in a relational database, semi-structured event logs, and unstructured documents that need chunking and embedding.

ETL (Extract, Transform, Load) and the more modern ELT (Extract, Load, Transform) differ in where transformation happens. ETL transforms data before loading it into the destination warehouse, useful when the destination has limited compute or when transformed data needs strict validation before storage. ELT loads raw data first and transforms it inside the warehouse using the warehouse's own compute (common with modern cloud warehouses like BigQuery/Snowflake, which have abundant, elastically scalable compute), which is more flexible since raw data remains available for reprocessing if transformation logic needs to change later.

Batch processing (e.g., Spark, Airflow-scheduled jobs) processes data in large, scheduled chunks and is well suited to workloads that don't need up-to-the-minute freshness. Stream processing (e.g., Kafka, Flink, Spark Streaming) processes data continuously as it arrives, event by event or in small micro-batches, and is required when a system needs near-real-time signal — such as computing a user's "actions in the last 5 minutes" feature for a fraud model.

Data validation and schema enforcement (tools like Great Expectations, or schema registries in a streaming pipeline) catch data quality problems at the pipeline level — a null rate suddenly spiking, a categorical field getting a new unexpected value, a numeric field's range shifting — before that bad data silently degrades a downstream model. This is the data engineering analog of the input validation discussed for individual queries in a RAG pipeline, just applied to entire datasets and pipelines rather than one request at a time.

A common interview question: "Your daily training data ingestion job succeeded with a green checkmark, but the resulting model's offline metrics tanked. What do you check first?" Answer: a successful pipeline run only confirms the job didn't crash, not that the data is correct — check row counts against historical norms, null rates per column, and schema/type consistency first, since a silent upstream schema change or a partial data load (e.g., a source system returning 10% of expected rows without erroring) is a far more common cause of this pattern than an actual, sudden shift in the real-world data distribution.
